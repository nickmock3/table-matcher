# 用語集（人間向け）

このプロジェクトで使う技術用語の簡易説明です。  
「一般的な意味」と「このプロジェクトでの使い方」を分けて記載します。

## Vertical Slice Architecture
- 一般的な意味:
  - 画面・API・ロジック・データアクセスを「機能単位」で縦にまとめる設計方針。
- このプロジェクトでの使い方:
  - `src/features/<feature>/...` を基本に、機能ごとにコードをまとめる。

## トランザクションスクリプト（Transaction Script）
- 一般的な意味:
  - 1つのユースケースを手続き的に実装するシンプルなスタイル。
- このプロジェクトでの使い方:
  - 初期実装はこの方式を基本にし、複雑化した機能だけ設計を昇格する。

## DDD（Domain-Driven Design）
- 一般的な意味:
  - 業務ルール（ドメイン）を中心にモデル設計・実装するアプローチ。
- このプロジェクトでの使い方:
  - 状態遷移や業務ルールが複雑な機能に限定して適用する。

## CQRS（Command Query Responsibility Segregation）
- 一般的な意味:
  - 更新（Command）と参照（Query）の責務を分離する設計。
- このプロジェクトでの使い方:
  - 読み取りと更新の関心が明確に異なる機能に限定して適用する。

## DoD（Definition of Done）
- 一般的な意味:
  - タスク/機能を「完了」と判定するための共通基準。
- このプロジェクトでの使い方:
  - `spec + plan + tasks` の整合、必要な検証、AIレビュー証跡を満たした状態を完了とする。

## AIレビュー
- 一般的な意味:
  - AIエージェントによるコードレビューや仕様整合チェック。
- このプロジェクトでの使い方:
  - 全PRで必須の品質ゲート。最終判断は開発者が行う。

## Hotfix
- 一般的な意味:
  - 緊急度の高い不具合に対する優先修正。
- このプロジェクトでの使い方:
  - 通常フローより優先するが、事後の記録・フォローアップを必須とする。
